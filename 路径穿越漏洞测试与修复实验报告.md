## 一、功能设计与构建

### 1.1 需求背景

本系统已经具备了登录认证、用户注册、信息搜索、头像上传、个人中心和余额充值等六个核心功能模块。随着功能增多，用户在使用过程中可能会遇到各种问题，例如不知道如何修改密码、不清楚充值上限是多少、不了解头像上支持的图片格式等等。为了提升用户体验，我们需要在系统中增加一个**帮助中心**页面，集中展示常见问题的答案和操作指南。

更进一步思考，未来可能还需要增加"关于我们"、"用户协议"、"隐私政策"、"版本更新日志"等更多静态内容页面。如果每增加一个页面就要去修改 Python 代码、注册新路由、创建新模板，不仅开发效率低，而且维护成本高。因此，我们决定设计一个**通用的动态页面加载机制**：通过统一的 URL 参数来控制加载哪个页面文件，路由代码只写一次，之后只需往指定目录投放 HTML 文件即可新增页面。

### 1.2 架构设计

整个动态页面加载系统的核心设计理念是"参数化文件加载"。用户在浏览器的 URL 中追加 `?name=xxx` 参数（例如 `/page?name=help`），Flask 后端解析出 `name` 的值，用它来定位服务器上的一个 HTML 文件，读取该文件的内容，然后把 HTML 片段注入到首页模板的指定区域中进行渲染。这样一来，前端开发者只需要关注 HTML 内容本身，完全不用改动后端代码。

```
用户浏览器请求: GET /page?name=help
        │
        ▼
┌───────────────────┐
│  Flask /page 路由  │  提取 name 参数 → name = "help"
└───────┬───────────┘
        │
        ▼
┌───────────────────────────────┐
│ os.path.join("pages", name)   │  构建路径 → pages/help
│ os.path.isfile(path)          │  检查文件是否存在
│ open(path).read()             │  读取文件内容
└───────┬───────────────────────┘
        │
        ▼
┌───────────────────┐
│ 返回 index.html    │  将 HTML 内容注入 page_content 变量
│  {{ page_content|safe }}   │  Jinja2 渲染到首页卡片区域
└───────────────────┘
```

值得注意的是，我们选择使用 Jinja2 的 `|safe` 过滤器来渲染页面内容。这意味着 HTML 标签不会被转义，页面可以直接包含 `<h2>`、`<table>`、`<img>` 等丰富的 HTML 元素。这为帮助中心页面提供了良好的排版能力，但同时也意味着——如果某个页面的内容被攻击者控制，恶意脚本将会直接在用户浏览器中执行。

### 1.3 路由设计

系统中新增的路由非常简洁，只有一个 `/page` 路由，通过 `name` 查询参数驱动：

| 路由 | HTTP 方法 | 参数 | 功能描述 |
|------|-----------|------|---------|
| `/page` | GET | `name` (必填) | 根据参数值定位并加载 pages/ 目录下的 HTML 文件 |
| `/page?name=help` | GET | name=help | 加载 pages/help.html |
| `/page?name=help.html` | GET | name=help.html | 同上（显式给出后缀） |
| `/page?name=about` | GET | name=about | 如果 pages/about 存在就加载它，否则尝试 pages/about.html |

路由设计了三个处理分支：
1. **name 为空** → 立即返回 "缺少 name 参数" 400 错误
2. **name 直接命中文件** → 读取内容并返回
3. **name 未命中** → 自动追加 `.html` 后缀再尝试一次，仍不存在则返回 "页面不存在"

这个"自动补全 `.html` 后缀"的设计虽然提升了用户体验（用户不需要记住文件后缀），但也在后面成为了一个额外的攻击面——补全后的路径同样可能产生穿越。

### 1.4 文件结构

为了实现这个功能，我们需要在项目中新增以下文件和目录：

```
Class01/                        ← 项目根目录
├── app.py                      ← 新增 /page 路由（约 25 行代码）
├── pages/                       ← 新建的页面目录
│   └── help.html                ← 帮助中心页面（纯 HTML 片段）
├── templates/
│   └── index.html               ← 在顶部新增 page_content 渲染区
└── static/css/style.css         ← 新增 .page-card 等动态页面样式
```

这里有一个关键设计决策：**pages/ 目录和 app.py 位于同一层级**（都在 Class01 项目根目录下）。这意味着 pages/ 的上层目录就是整个项目的根目录，其中包含了 `.secret_key`（Flask 的 session 签名密钥）、`app.py`（完整源码）、`requirements.txt`（依赖清单）、`data/users.db`（SQLite 数据库文件）等大量敏感文件。这个看似无害的目录结构，成为了后续路径穿越攻击的地理基础——攻击者只需要向上跳一级就能碰到这些敏感文件。

### 1.5 初始代码实现（漏洞版本）

以下是我们按照需求规格编写的第一版代码。它的逻辑清晰、功能正确——只要你对 `os.path.join` 的行为不够了解，完全看不出这短短的 20 行代码有什么问题：

```python
@app.route("/page")
def page():
    """动态页面加载 —— name 直接拼接路径，不校验 ../"""
    name = request.args.get("name", "")

    if not name:
        return "缺少 name 参数", 400

    # 直接拼接用户输入的 name 到路径中
    page_path = os.path.join("pages", name)

    # 如果文件存在则读取内容
    if os.path.isfile(page_path):
        with open(page_path, "r", encoding="utf-8") as f:
            content = f.read()
        return render_template("index.html", page_content=content)

    # 不存在 → 尝试加上 .html 后缀
    page_path_html = os.path.join("pages", name + ".html")
    if os.path.isfile(page_path_html):
        with open(page_path_html, "r", encoding="utf-8") as f:
            content = f.read()
        return render_template("index.html", page_content=content)

    return "页面不存在"
```

这段代码的设计意图非常直白：
1. 从 URL 参数获取页面名称
2. 用 `os.path.join` 拼接 `pages/` 和用户输入的 `name`
3. 检查这个路径指向的文件是否存在
4. 如果存在，用 `open()` 读取并返回内容
5. 如果不存在，给 `name` 追加 `.html` 后缀再试

我在这里特别强调：这段代码**完美实现了需求规格中的所有功能**。`/page?name=help` 能正常加载帮助中心，`/page?name=about` 会正确显示"页面不存在"。代码简洁、功能正确——但正是这份简洁掩盖了致命的安全缺陷。

### 1.7 前端模板和样式修改

在 `templates/index.html` 中，我们在页面最顶部新增了一个条件渲染区块。当 `page_content` 变量存在时（即用户通过 `/page?name=xxx` 访问时），页面会先渲染一个卡片区域来展示动态加载的 HTML 内容，然后再显示用户信息卡片和搜索卡片：

```html
{% if page_content %}
<div class="card page-card">
    {{ page_content|safe }}
</div>
{% endif %}
```

这里使用了 Jinja2 的 `|safe` 过滤器，它的作用是不对变量内容进行 HTML 转义。换句话说，`page_content` 中的 `<h2>`、`<table>` 等标签会被浏览器直接解析为 HTML 元素，而不是显示为 `&lt;h2&gt;` 这样的纯文本。这是既能展示丰富排版的关键——也是潜在 XSS 风险的来源。

同时，在已登录用户的操作按钮区域，我们增加了一个"帮助中心"快捷入口：

```html
<a href="/page?name=help" class="btn btn-secondary btn-sm">帮助中心</a>
```

对应的 CSS 样式为 `.page-card` 增加了左侧蓝色边框、适当的内边距和标题层次样式，使其在视觉上能与用户信息卡片有效区分。

## 二、漏洞原理分析

### 2.1 路径穿越的核心机制 —— os.path.join 的秘密行为

Python 标准库的 `os.path.join` 函数的设计目的是"智能路径拼接"，而不是"安全路径限制"。文档中明确描述了一个非常关键的行为：**如果第二个参数以 `/` 或 `\` 开头，第一个参数会被完全忽略**；**如果第二个参数包含 `..` 组件，它会被正常解析为父目录引用**。

以下是用实际代码演示的效果：

```python
# 正常情况 —— 预期的行为
os.path.join("pages", "help")              → "pages/help"
os.path.join("pages", "help.html")         → "pages/help.html"

# 路径穿越 —— 攻击者利用的行为
os.path.join("pages", "../.secret_key")     → ".secret_key"
# ↑ "pages" + "../.secret_key" = "pages/../.secret_key" = ".secret_key"
#   即 pages 的上层目录（项目根目录）下的 .secret_key 文件

os.path.join("pages", "../../etc/passwd")  → "../../etc/passwd"
# ↑ 向上跳两级，到达文件系统的 /etc/ 目录

os.path.join("pages", "../app.py")         → "app.py"
# ↑ 向上跳一级，到达项目根目录的 app.py

os.path.join("pages", "/etc/passwd")       → "/etc/passwd"
# ↑ 绝对路径的情况：第二个参数以 / 开头，第一个参数 pages 被完全丢弃
```

这个行为本身是 Python 路径操作的标准特性，在绝大多数场景下是完全合理且有用的。问题在于：**我们的代码将用户可控的数据（name 参数）直接传给了 os.path.join 的第二个参数位置**，而没有对 `../` 或 `/` 做任何过滤。这使得攻击者可以自由地指定任意文件路径。

### 2.2 完整的攻击链路分析

现在让我们追踪一个攻击者从访问正常功能到窃取核心机密的完整过程。攻击者首先正常访问帮助中心确认功能可用：

**第一步：侦察**  
攻击者访问 `/page?name=help`，页面正常显示帮助中心内容。这告诉攻击者两点关键信息：(1) `/page` 路由确实在读取并显示文件内容；(2) 文件读取使用的是与服务端代码相同的文件系统权限。这两点都暗示了路径穿越的可能性。

**第二步：探测路径结构**  
攻击者尝试 `/page?name=../app.py`。根据我们对 `os.path.join` 行为的分析，这会生成路径 `app.py`——也就是项目根目录下的主程序文件。如果攻击者注意到页面返回的不是 "页面不存在" 而是大段 Python 代码，他就知道路径穿越成功了。

**第三步：纵深利用**  
得到 `app.py` 的完整源码后，攻击者可以：
- 找到 `SECRET_KEY_FILE` 配置项 → 发现 `.secret_key` 文件的存在
- 用 `../.secret_key` 读取密钥 → 获得 Flask session 签名密钥
- 阅读 `/register` 路由 → 了解注册接口的请求格式
- 阅读 `/recharge` 路由 → 发现充值的越权漏洞
- 阅读 `/search` 路由 → 发现 SQL 注入点
- 阅读 `data/users.db` 路径 → 用 `../data/users.db` 直接下载数据库文件

**第四步：完整控制**  
有了 session 签名密钥，攻击者可以伪造任意用户的 Flask session cookie，以任意用户身份登录系统（包括 admin）。知道了源码中的所有漏洞，攻击者可以依次利用——水平越权查看所有用户资料、SQL 注入窃取完整用户表、负数充值刷钱、甚至通过路径穿越读取服务器上的其他敏感配置文件。

这整个过程完全不需要登录，不需要任何前置权限，甚至不需要复杂的工具——只需要一个浏览器，手动修改 URL 参数即可。

### 2.3 各类敏感文件泄露的危害分析

下面我们逐一分析通过路径穿越可能泄露的每一类文件，及其被泄露后造成的实际危害：

| 目标文件 | 穿越载荷 | 泄露内容 | 连锁危害 |
|---------|---------|---------|---------|
| `.secret_key` | `../.secret_key` | Flask session 签名密钥（明文 48 位十六进制字符串） | 攻击者可伪造任意用户的 session cookie，包括 admin 账户。一旦获得密钥，整个 session 认证体系形同虚设 |
| `app.py` | `../app.py` | 完整的 Flask 应用源码（约 500 行），包含所有路由逻辑、数据库查询、密钥管理、安全配置 | 攻击者获得完整的攻击面地图：知道每个路由的实现细节，可以针对性地构造攻击 payload |
| `requirements.txt` | `../requirements.txt` | 依赖库名称和版本号（Flask>=3.0, Flask-WTF>=1.2 等） | 根据依赖版本查找已知 CVE 漏洞，例如特定版本的 Werkzeug 或 Jinja2 的公开 RCE |
| `data/users.db` | `../data/users.db` | SQLite 数据库文件，含所有注册用户的用户名、密码哈希、邮箱、手机号、余额 | 用户隐私完全泄露；虽然密码是 PBKDF2 哈希，但仍可离线暴力破解；余额信息可被用于精准诈骗 |
| `templates/login.html` | `../templates/login` | 登录页面的 HTML 源码，包括表单结构、CSRF token 名称和位置 | 了解防御机制的具体实现细节，为 CSRF 或自动化攻击提供精确模板 |
| `templates/register.html` | `../templates/register` | 注册页面的 HTML 源码，包括 CSRF token 结构 | 可编写自动化注册脚本，批量创建虚假账户 |
| `../../etc/passwd` | `../../etc/passwd` | Linux 系统用户列表（用户名、UID、GID、Shell 等） | 识别系统中的有效用户名，为进一步的 SSH 爆破提供目标清单 |
| `../../etc/shadow` | `../../etc/shadow` | 系统用户密码哈希文件（通常需要 root 权限才能读取，但值得尝试） | 如果 Flask 进程以 root 运行，此文件泄露将直接导致服务器完全沦陷 |

### 2.4 攻击面矩阵分析

我们将所有可能的攻击向量按穿越层数和利用难度进行分类：

| 攻击向量分类 | 载荷示例 | 穿越层数 | 利用难度 | 危害描述 |
|-------------|---------|---------|---------|---------|
| 基本穿越（一级） | `../.secret_key` | 1 级（pages/ → 项目根目录） | 极低 | 只需知道目标文件名，零编码 |
| 基本穿越（一级） | `../app.py` | 1 级 | 极低 | 系统主文件，100% 存在 |
| 基本穿越（一级） | `../requirements.txt` | 1 级 | 极低 | 依赖清单，100% 存在 |
| .html 补全穿越 | `../templates/login` | 1 级 + 自动补全 | 低 | 利用自动补全机制读取模板 |
| .html 补全穿越 | `../templates/index` | 1 级 + 自动补全 | 低 | 同上，读取首页模板 |
| 多级穿越 | `../../etc/passwd` | 2 级 | 低 | 读取系统级文件 |
| 多级穿越 | `../../etc/hostname` | 2 级 | 低 | 获取主机名 |
| 绝对路径穿越 | `/etc/passwd` | os.path.join 优先级 | 低 | 第一个参数被完全忽略 |
| 编码绕过 | `..%2F..%2Fetc%2Fpasswd` | URL 编码后 | 中 | 如果代码做了 `/` 的字符串过滤，但没做 URL 解码 |
| 双重编码 | `..%252F..%252Fetc%252Fpasswd` | 双重 URL 编码 | 高 | 绕过做了单层 URL 解码的过滤器 |

### 2.5 为什么简单的字符串过滤不够？

很多开发者第一反应是："我加个判断，如果 name 包含 `../` 就拒绝不就好了？" 但实际上，字符串黑名单过滤存在大量绕过方式：

- **URL 编码绕过**：`../` 的 URL 编码形式是 `%2E%2E%2F` 或 `..%2F`，如果过滤器在 URL 解码之前执行，编码后的 payload 不会匹配到 `../` 字符串
- **绝对路径绕过**：`/etc/passwd` 不包含任何 `../`，但 `os.path.join("pages", "/etc/passwd")` 的结果就是 `/etc/passwd`——完全不需要穿越
- **长路径探测**：`....//....//etc/passwd` 在某些操作系统的路径归一化处理后等价于 `../../etc/passwd`
- **Unicode 规范化绕过**：`..∕` 在某些系统中会被规范化为 `../`，但字符串比较检测不到

因此，安全行业的最佳实践从来不是"过滤危险字符"，而是**限定允许的范围**——即白名单思维而不是黑名单思维。

#### POC 2：路径穿越读取 Flask 密钥

## 四、修复过程

### 4.1 安全修补策略

经过对漏洞原理的深入分析，路径穿越问题的根源非常清晰：**用户输入的 name 参数直接控制着 `os.path.join` 的结果，而不受任何边界限制**。这意味着 `os.path.join` 生成的路径可以在整个文件系统中自由移动。

因此，修复需要解决两个核心问题：
1. **路径结果不可控** → 需要将解析后的路径限制在 pages/ 目录的安全范围内
2. **`../` / 绝对路径可以绕过目录限制** → 不能依赖字符串过滤，需要文件系统级别的路径规范化

基于以上分析，我们设计了"**真实路径解析 + 前缀安全比对**"的修复方案。这个方案分为四个步骤：

| 步骤 | 技术手段 | 解决的问题 |
|------|---------|-----------|
| **第一层：基准路径固化** | `PAGES_DIR = os.path.realpath("pages")` | 获取 pages/ 目录的绝对真实路径，消除符号链接、相对路径等干扰因素 |
| **第二层：输入清洗** | `name.replace("\\", "/").lstrip("/")` | 统一路径分隔符格式，去除开头的 `/`（因为 `os.path.join` 对绝对路径会丢弃第一个参数，必须阻止这种情况） |
| **第三层：真实路径解析** | `os.path.realpath(os.path.join(PAGES_DIR, name))` | 将拼接后的路径解析为真实的文件系统路径，彻底消解 `../` 等相对路径组件 |
| **第四层：安全边界校验** | `resolved.startswith(PAGES_DIR + os.sep)` | 确认解析后的路径确实位于 pages/ 目录树内，任何穿越都会在此时被检测到并拒绝 |

为什么选择 `os.path.realpath` 而不是 `os.path.abspath`？  
`os.path.abspath` 只是将相对路径转为绝对路径，不会解析符号链接，也不会消除 `..` 组件。例如 `os.path.abspath("pages/../../etc/passwd")` → `/etc/passwd`，它仍然会追踪到 pages/ 之外。而 `os.path.realpath` 在解析 `..` 的同时还会处理符号链接，给出文件的实际存储位置。

### 4.2 修复前后代码详细对比

#### ❌ 漏洞版本（修复前）

```python
@app.route("/page")
def page():
    """动态页面加载 —— name 直接拼接路径，不校验 ../"""
    name = request.args.get("name", "")

    if not name:
        return "缺少 name 参数", 400

    # ⚠️ 问题1：用户输入的 name 直接传给 os.path.join 第二个参数
    #    name = "../.secret_key"
    #    → os.path.join("pages", "../.secret_key")
    #    → ".secret_key"（穿越到项目根目录！）
    page_path = os.path.join("pages", name)

    # ⚠️ 问题2：没有任何路径范围校验，只要文件存在就读取
    if os.path.isfile(page_path):
        with open(page_path, "r", encoding="utf-8") as f:
            content = f.read()
        return render_template("index.html", page_content=content)

    # ⚠️ 问题3：.html 补全路径同样未校验
    #    name = "../templates/login"
    #    → os.path.join("pages", "../templates/login.html")
    #    → "templates/login.html"（穿越到 templates 目录！）

    if os.path.isfile(page_path_html):

        return render_template("index.html", page_content=content)

```

漏洞版本的根本问题是**完全依赖用户输入来确定文件路径**，缺少一个独立的、不受用户控制的边界限制机制。

#### ✅ 安全版本（修复后）

```python
# 第一步：在函数外部获取 pages/ 的绝对真实路径作为基准
# 使用 os.path.realpath 解析所有符号链接和 ../
# 这样得到的是一个固定的、不可篡改的锚点
PAGES_DIR = os.path.realpath("pages")


@app.route("/page")
def page() -> str:
    """动态页面加载 —— 路径穿越已修复"""
    name = request.args.get("name", "")

    if not name:
        return "缺少 name 参数", 400

    # 第二步：清洗用户输入
    # 统一反斜杠为正斜杠（跨平台兼容）
    # 去除开头的正斜杠——因为 os.path.join 对以 / 开头的第二参数
    # 会完全丢弃第一参数，必须阻止这种情况
    name = name.replace("\\", "/")
    name = name.lstrip("/")

    # 第三步：拼接并解析为真实文件系统路径
    # os.path.realpath 会将 ../ 解析为实际的父目录路径
    # 例如 join(PAGES_DIR, "../app.py") → realpath → "/opt/Class01/app.py"
    page_path = os.path.join(PAGES_DIR, name)
    page_path = os.path.realpath(page_path)

    # 第四步：核心安全检查 —— 解析后的路径必须在 pages/ 目录内
    # PAGES_DIR + os.sep 确保匹配的是目录级别而非字符串前缀
    # 例如: /opt/Class01/pages 只匹配 /opt/Class01/pages/xxx
    #      不会误匹配 /opt/Class01/pages_backup/xxx
    if not page_path.startswith(PAGES_DIR + os.sep):
        return "页面不存在"

    # 第五步：文件存在即安全读取（此时路径已通过边界校验）
    if os.path.isfile(page_path):
        with open(page_path, "r", encoding="utf-8") as f:
            content = f.read()
        return render_template("index.html", page_content=content)

    # 第六步：.html 后缀补全分支同样需要进行边界校验
    page_path_html = os.path.join(PAGES_DIR, name + ".html")
    page_path_html = os.path.realpath(page_path_html)
    if not page_path_html.startswith(PAGES_DIR + os.sep):
        return "页面不存在"

    if os.path.isfile(page_path_html):
        with open(page_path_html, "r", encoding="utf-8") as f:
            content = f.read()
        return render_template("index.html", page_content=content)

    return "页面不存在"
```

以下通过实际路径推演来说明修复前后的差异。假设 `PAGES_DIR = os.path.realpath("pages")` 计算出的基准路径是 `/opt/Class01/pages`：

**场景一：正常请求 `name=help`**

```

        → "/opt/Class01/.secret_key"   ← realpath 解析了 ../
```

**场景四：绝对路径 `name=/etc/passwd`**

## 七、安全设计原则总结

通过本次路径穿越漏洞的完整生命周期——从设计构建、到漏洞发现、到 exploit 验证、再到安全修复——我们可以提炼出以下核心安全原则：

1. **永远不要信任用户提供的文件路径参数**。在本次实验中，一个看似无害的 `name` 查询参数，未经任何处理就能读取服务器上几乎任何文本文件。任何来自客户端的路径参数都必须被视为潜在的恶意输入。

2. **`os.path.join` 不是安全函数**。Python 文档明确指出它的作用是智能拼接路径组件，而不是限制路径范围。`../` 在它的语义中是合法的父目录引用，会被正常解析。将其用于含有用户输入的路径拼接时，必须额外添加边界校验。

3. **黑名单过滤是不可靠的**。简单的 `if "../" in name` 过滤可以被 URL 编码、绝对路径、Unicode 规范化等多种方式绕过。正确的做法是限定合法范围（白名单思维）：定义好安全目录的边界，然后验证解析后的路径是否落在边界之内。

4. **`os.path.realpath` + `startswith` 前缀比对是路径穿越的标准防御模式**。这个组合能可靠地检测任何形式的路径穿越（包括 `../`、符号链接、绝对路径、编码绕过），因为它直接在文件系统层面计算真实位置并与合法范围比较，不依赖于字符串层面的模式匹配。

5. **不存在的文件和被拒绝的请求应该返回相同的错误信息**。在本次修复中，我们把所有被拒绝的请求都返回"页面不存在"（与文件真实不存在时相同的提示），避免通过响应差异向攻击者泄露信息。

6. **纵深防御：即使一个环节的校验失效，其他环节仍能兜底**。在我们的四步修复中，`lstrip("/")` 是第一道防线（阻止绝对路径），`os.path.realpath` 是第二道防线（解析相对路径），`startswith` 是第三道防线（终极边界守卫）。三层之间互相独立，任何一层单独工作都能防止穿越。
