## 一、功能设计与构建

### 1.1 需求背景

在已有系统（登录 / 注册 / 搜索 / 头像上传 / 个人中心 / 充值）基础上，新增**动态页面加载功能**：通过 URL 参数指定页面名称，服务端动态读取并渲染对应的 HTML 内容到首页。

### 1.2 架构设计

```
用户请求: /page?name=help
           │
     ┌─────▼─────┐
     │  Flask 路由 │
     │  /page     │
     └─────┬─────┘
           │ 提取 name 参数
     ┌─────▼─────────────────────────┐
     │ os.path.join("pages", name)    │
     │ 拼接路径 → 读取 HTML 文件       │
     └─────┬─────────────────────────┘
           │
     ┌─────▼─────┐
     │ 首页渲染   │
     │ index.html│ ← page_content 展示区
     └───────────┘
```

### 1.3 路由设计

| 路由 | 方法 | 参数 | 功能 |
|------|------|------|------|
| `/page` | GET | `name` | 动态加载 `pages/` 目录下的 HTML 页面 |
| — | — | `name=help` | 加载 `pages/help.html` |
| — | — | `name=help.html` | 同上（自动补全后缀） |

### 1.4 文件结构

```
Class01/
├── app.py              ← 新增 /page 路由
├── pages/               ← 新建目录
│   └── help.html        ← 帮助中心页面
├── templates/
│   └── index.html       ← 新增 page_content|safe 展示区 + 帮助中心入口
└── static/css/style.css ← 新增 .page-card 样式
```

### 1.5 初始代码（漏洞版本）

```python
@app.route("/page")
def page():
    name = request.args.get("name", "")
    if not name:
        return "缺少 name 参数", 400

    # 直接拼接用户输入的 name，不校验 ../
    page_path = os.path.join("pages", name)

    if os.path.isfile(page_path):
        content = open(page_path).read()
        return render_template("index.html", page_content=content)

    # 不存在 → 自动加 .html 后缀
    page_path_html = os.path.join("pages", name + ".html")
    if os.path.isfile(page_path_html):
        content = open(page_path_html).read()
        return render_template("index.html", page_content=content)

    return "页面不存在"
```

### 1.6 help.html 内容

帮助中心静态页面，包含常见问题解答和联系方式：

## 二、漏洞分析

### 2.1 漏洞原理 — 路径穿越（Path Traversal / Directory Traversal）

`os.path.join("pages", name)` 有一个关键行为：当 `name` 以 `/` 或包含 `../` 时，`os.path.join` 会将路径解析到上层目录。

```python
os.path.join("pages", "help")           → pages/help
os.path.join("pages", "../.secret_key")  → .secret_key        ← 上一级目录！
os.path.join("pages", "../../etc/passwd")→ ../../etc/passwd   ← 上两级！
os.path.join("pages", "../app.py")       → app.py             ← 项目根目录！
```

**漏洞链**：
```
攻击者输入 name=../app.py
  → os.path.join("pages", "../app.py") → "app.py"
  → os.path.isfile("app.py") → True（app.py 就在项目根目录）
  → open("app.py").read() → 返回完整源码
  → 渲染到页面 → 攻击者获得源码
```

### 2.2 为什么如此危险？

| 泄露文件 | 危害 |
|---------|------|
| `../.secret_key` | Session 签名密钥泄露 → 可伪造任意用户 Session |
| `../app.py` | 完整源码泄露 → 暴露所有内部逻辑和漏洞 |
| `../requirements.txt` | 依赖清单 → 知道使用了哪些库和版本，可针对性攻击 |
| `../data/users.db` | 用户数据库 → 含密码哈希和用户信息 |
| `../templates/login` | 模板源码 → 了解表单结构和 CSRF 实现 |
| `../../etc/passwd` | 系统用户文件 → 系统级信息泄露 |

### 2.3 攻击面分析

| 攻击向量 | 载荷 | 原理 |
|---------|------|------|
| 基本穿越 | `../.secret_key` | 返回上一级目录的文件 |
| 多级穿越 | `../../etc/passwd` | 连续返回多层目录 |
| 自动补全利用 | `../templates/login` | 借助 .html 补全读取 login.html |
| 绝对路径 | `/etc/passwd` | os.path.join 忽略 pages/，直接拼接绝对路径 |

## 四、修复过程

### 4.1 修复策略

路径穿越的根本问题是：**攻击者可以通过 `../` 使路径解析到目标目录之外**。

修复方案：**使用 `os.path.realpath()` 解析真实路径后用前缀比对限制范围**。

| 步骤 | 做法 | 作用 |
|------|------|------|
| 1 | `os.path.realpath("pages")` 获取 pages/ 的真实绝对路径 | 作为合法范围基准 |
| 2 | `name.replace("\\", "/").lstrip("/")` | 去除 Windows 反斜杠和开头正斜杠 |
| 3 | `os.path.realpath(os.path.join(PAGES_DIR, name))` | 拼接后解析为真实路径 |
| 4 | `resolved_path.startswith(PAGES_DIR + os.sep)` | 确认解析后的路径仍在 pages/ 内 |

**关键原理**：`os.path.realpath("../.secret_key")` 会计算出这个路径的真实位置（即项目的 `.secret_key`），然后检查它是否以 `pages/` 的真实路径开头。如果不是 → 拒绝。

### 4.2 修复前后代码对比

#### ❌ 修复前

```python
@app.route("/page")
def page():
    name = request.args.get("name", "")
    # 直接拼接，../ 会穿越到上层目录
    page_path = os.path.join("pages", name)

    if os.path.isfile(page_path):
        content = open(page_path).read()
        return render_template("index.html", page_content=content)
```

#### ✅ 修复后

```python
PAGES_DIR = os.path.realpath("pages")   # 基准真实路径

@app.route("/page")
def page():
    name = request.args.get("name", "")
    # 1. 清洗输入 → 去除开头的 / 和 \\
    name = name.replace("\\", "/").lstrip("/")
    # 2. 拼接并解析真实路径
    page_path = os.path.realpath(os.path.join(PAGES_DIR, name))
    # 3. 边界校验 → 必须在 pages/ 内
    if not page_path.startswith(PAGES_DIR + os.sep):
        return "页面不存在"
    # 4. 安全读取
    if os.path.isfile(page_path):
        content = open(page_path).read()
        return render_template("index.html", page_content=content)
```

### 4.3 修复效果分析

```
用户输入: name=help
  → os.path.realpath("/opt/Class01/pages/help.html")
  → 以 /opt/Class01/pages/ 开头 ✅ 允许

用户输入: name=../.secret_key
  → os.path.realpath("/opt/Class01/.secret_key")
  → 不以 /opt/Class01/pages/ 开头 ❌ 拒绝

用户输入: name=../app.py
  → os.path.realpath("/opt/Class01/app.py")
  → 不以 /opt/Class01/pages/ 开头 ❌ 拒绝

用户输入: name=/etc/passwd
  → os.path.realpath("/etc/passwd")
  → 不以 /opt/Class01/pages/ 开头 ❌ 拒绝
```

## 七、安全设计原则

1. **永远不要信任用户提供的路径参数** — 单个 `name` 可以变成任意系统文件路径
2. **os.path.join 不防穿越** — 它是拼接工具，不是安全工具，`../` 会正常解析
3. **os.path.realpath + 前缀比对** 是标准防御模式 — 先解析真实路径，再确认在合法范围内
4. **自动补全 .html 也要校验** — 补全后的路径同样可能产生穿越
5. **编码绕过也需防范** — 只做字符串替换不够，`realpath` 能处理 `..` 语义层面
