```
└─────────────┴─────────────┴──────────────┴───────────────┘
```

### 1.2 上传功能设计

| 路由 | 方法 | 权限 | 功能 |
|------|------|------|------|
| `/upload` | GET | 需登录 | 显示上传页面 |
| `/upload` | POST | 需登录 | 接收文件，校验后保存 |

### 1.3 初始代码（漏洞版本）

```python
@app.route("/upload", methods=["GET", "POST"])
@csrf.exempt
def upload():
    if request.method == "POST":
        f = request.files["file"]
        save_path = os.path.join(UPLOAD_DIR, f.filename)  # ← 直接使用用户提供的文件名
        f.save(save_path)
```

**漏洞清单**：
1. ❌ 不检查文件扩展名 — 任意后缀均可上传
2. ❌ 不检查 MIME 类型 — 不验证文件内容
3. ❌ 使用原始文件名保存 — 路径穿越 / 覆盖
4. ❌ 无大小限制检查 — 超大文件耗尽磁盘
5. ❌ CSRF 豁免无保护 — 缺少纵深防御

## 二、漏洞原理分析

### 2.1 为什么文件上传漏洞如此危险？

文件上传功能允许用户将文件保存到服务器。如果不对上传的文件做**后缀校验**、**内容校验**和**文件名清洗**，攻击者可以：

1. 上传 `.php` / `.asp` / `.jsp` 等可执行脚本 → **Webshell** → 服务器沦陷
2. 上传 `.html` 文件 → **存储型 XSS** / 钓鱼页面
3. 通过 `../` 路径穿越 → **覆盖系统文件**（如 SSH 密钥、cron 任务）
4. 上传恶意 JavaScript → **第三方脚本劫持**
5. 同名文件覆盖 → **已有文件被篡改**

### 2.2 Webshell 上传绕过技术原理

#### 2.2.1 黑名单后缀的局限性

黑名单位于维护成本高、易绕过。常见绕过方式：

| 绕过方式 | 绕过原理 | 示例 |
|---------|---------|------|
| **大小写绕过** | 黑名单只检查小写 `.php` | `.PHP` `.PhP` `.pHP` |
| **双写绕过** | 黑名单将 `php` 替换为空 → `pphphp` → `php` | `.pphphp` |
| **点号绕过** | Windows 会自动去除尾部 `.` | `.php.` → `.php` |
| **空格绕过** | Windows 会自动去除尾部空格 | `.php ` → `.php` |
| **组合绕过** | Windows 会去除尾部 ` .` | `.php. .` → `.php` |
| **::$DATA 绕过** | NTFS 备用数据流在解析时被忽略 | `.php::$DATA` → `.php` |
| **%00 截断** | 老版本 PHP 在 `%00` 处截断 | `.php%00.jpg` → `.php` |
| **路径穿越** | `../` 跳出上传目录 | `../../etc/cron.d/shell.php` |

#### 2.2.2 为什么需要白名单？

白名单**只允许**特定的扩展名（如 `.jpg` / `.png` / `.gif`），不在列表中的一律拒绝。这从根本上杜绝了上述所有绕过：

```
黑名单方式：   拒绝 .php → 但 .PHP / .PhP / .php5 / .phtml 仍能通过
白名单方式：   只允许 .jpg →  .php / .PHP / .asp 统统不行
```

#### 2.2.3 为什么需要内容校验？

仅仅检查扩展名不够——攻击者可以上传一个**真实图片**，但在图片的 EXIF 注释或文件尾部嵌入 PHP 代码（图片马）：

```
图片马：
  \x89PNG\r\n\x1a\n  ← 真实 PNG 文件头
  ...
  <?php system($_GET["cmd"]); ?>  ← 嵌入 PHP 代码（被 Apache 忽略，被包含时执行）
```

即使直接上传 `.jpg` 文件，也需要通过 **imghdr** 或 **python-magic** 验证文件头，确保内容与扩展名一致。

## 四、修复过程

### 4.1 修复策略

| 漏洞 | 修复方案 | 原理 |
|------|---------|------|
| 任意后缀上传 | **白名单校验** `ALLOWED_EXTENSIONS` | 只允许 jpg/jpeg/png/gif/webp |
| 大小写绕过 | **统一小写** `filename.lower()` | `.PHP` → `.php` → 被白名单拒绝 |
| 点号/空格绕过 | **去除尾部和点号** `.rstrip(". ")` | `shell.php.` → `shell.php` → 被白名单拒绝 |
| `::$DATA` 绕过 | **去除 `::` 之后内容** | `shell.php::$DATA` → `shell.php` → 被白名单拒绝 |
| 路径穿越 | **仅保留文件名最后一段** | `../../x.php` → `x.php` → 被白名单拒绝 |
| 内容伪造 | **imghdr 验证文件头** | `\x89PNG` → 有效 | `<?php` → 拒绝 |
| 文件覆盖 | **UUID 重命名** | `uuid4().hex.ext` → 不可预测、不重复 |
| 非法内容 | **白名单 + 内容校验双层防护** | 扩展名通过后仍需验证文件二进制头 |

### 4.2 修复前后代码对比

#### 修复前
```python
# 漏洞版本 — 无任何校验
UPLOAD_DIR = "static/uploads"

@app.route("/upload", methods=["GET", "POST"])
@csrf.exempt
def upload():
    f = request.files["file"]
    save_path = os.path.join(UPLOAD_DIR, f.filename)  # 直接使用原始文件名
    f.save(save_path)
```

#### 修复后
```python
# 安全版本 — 白名单 + 清洗 + 内容校验 + UUID 重命名
ALLOWED_EXTENSIONS = {"jpg", "jpeg", "png", "gif", "webp"}

def _sanitize_filename(filename: str) -> str:
    """清洗文件名，防止各种后缀绕过"""
    # 1. 去除路径穿越
    filename = filename.replace("\\", "/")
    filename = filename.split("/")[-1]
    # 2. 去除 NTFS 备用数据流
    if "::" in filename:
        filename = filename.split("::")[0]
    # 3. 去除 null 字节
    filename = filename.replace("\x00", "")
    # 4. 去除尾部空格和点号
    filename = filename.rstrip(". ")
    # 5. 转换为小写
    filename = filename.lower()
    return filename

def _is_allowed_file(filename: str) -> bool:
    """白名单校验文件扩展名"""
    if "." not in filename:
        return False
    ext = filename.rsplit(".", 1)[1].lower()
    return ext in ALLOWED_EXTENSIONS

@app.route("/upload", methods=["GET", "POST"])
def upload():
    # 1. 清洗文件名
    safe_filename = _sanitize_filename(f.filename)
    # 2. 白名单校验
    if not _is_allowed_file(safe_filename):
        return "仅允许上传图片文件"
    # 3. 内容校验
    file_data = f.read(512)
    detected = imghdr.what(None, file_data)
    if detected is None or detected not in ("jpeg", "png", "gif", "webp"):
        return "文件内容不是有效图片"
    # 4. UUID 重命名 + 保存
    new_name = f"{uuid.uuid4().hex}.{ext}"
    save_path = os.path.join(UPLOAD_DIR, new_name)
    f.save(save_path)
```

```python

# 用户上传 shell.PHP  → 清洗后: shell.php  → 白名单校验: ❌ 拒绝
# 用户上传 shell.php. → 清洗后: shell.php  → 白名单校验: ❌ 拒绝
# 用户上传 shell.php ::$DATA → 清洗后: shell.php → 白名单校验: ❌ 拒绝
```

## 七、安全编码原则总结

### 7.1 文件上传安全金三角

```
                    ┌─────────────┐
                    │  白名单校验   │
                    │  (扩展名)    │
                    └──────┬──────┘
                           │
            ┌──────────────┼──────────────┐
            │              │              │
     ┌──────▼──────┐ ┌─────▼──────┐ ┌─────▼──────┐
     │  内容校验    │ │  重命名     │ │  目录限制   │
     │ (imghdr)    │ │ (UUID)     │ │ (路径清洗)  │
     └─────────────┘ └────────────┘ └────────────┘
```

三者缺一不可：
- **白名单** 杜绝可执行脚本上传
- **内容校验** 防止图片马
- **重命名** 防止覆盖和路径穿越

### 7.2 核心知识点

| 概念 | 原理 |
|------|------|
| **黑名单 vs 白名单** | 黑名单列举"不允许的"，白名单列举"只允许的"——白名单更安全 |
| **后缀绕过原理** | 操作系统/Web 服务器在处理文件名时，可能自动去除尾部特殊字符 |
| **NTFS 备用数据流(ADS)** | Windows 文件可包含隐藏数据流 `file:stream`，IIS 解析时忽略 `::$DATA` |
| **imghdr 文件头验证** | 真实图片文件的前几个字节是固定的魔数（Magic Number），如 PNG = `\x89PNG` |
| **UUID 重命名** | 通用唯一标识符，不可预测的 32 位十六进制字符串，杜绝文件名碰撞 |

### 7.3 纵深防御建议

```
1️⃣ 前端校验 （用户体验，非安全措施）
    → 限制文件选择器为 image/*

2️⃣ 后端白名单 （核心安全屏障）
    → 只允许 jpg/png/gif/webp

3️⃣ 内容校验 （第二道屏障）
    → imghdr 确认文件头与扩展名一致

4️⃣ UUID 重命名 （防覆盖/穿越）
    → uuid4().hex 无规律文件名

5️⃣ 访问控制
    → 上传目录禁止执行脚本（.htaccess 或 Nginx 配置）
      `Options -ExecCGI` / `location ~ \.php { deny all; }`
```

## 八、总结

1. **不安全的文件上传是最高危漏洞之一**：一旦上传 Webshell，攻击者可执行任意系统命令
2. **黑名单永远不够**：`php` → `PHP` / `PhP` / `php5` / `phtml` / `php.` / `php::$DATA`……绕过方式无穷
3. **白名单是唯一正道**：明确只允许图片格式，不在列表中的一律拒绝
4. **文件名清洗必须多层**：大小写转换 + 去除特殊字符 + 去除路径穿越 = 单个函数解决 10 类绕过
5. **内容校验是第二道保险**：即使扩展名是 `.jpg`，也需验证文件头确认真是图片
6. **UUID 重命名一举多得**：防遍历、防覆盖、防可预测文件名攻击
