# 用户信息管理平台安全开发实践报告

> **项目名称**：用户信息管理平台  
> **版本迭代**：V1.0（基础版）→ V2.0（安全加固版）  
> **技术栈**：Python Flask + Jinja2 + Werkzeug + Flask-WTF + Flask-Limiter  
> **代码仓库**：https://github.com/Perry88882/SecondDayOfTraining  
> **开发环境**：Kali Linux / Python 3.13 / Flask 3.1

---

## 一、项目概述

### 1.1 需求说明

开发一个基于 Flask 的用户信息管理平台，实现用户登录、个人信息展示、登出等基础功能。

### 1.2 功能清单

| 模块 | 功能 |
|------|------|
| 登录 | 用户名/密码验证，Session 会话保持 |
| 首页 | 已登录展示用户信息，未登录提示跳转 |
| 登出 | 清除 Session 并重定向 |
| 导航栏 | 根据登录状态动态切换菜单 |

### 1.3 开发策略

采用"先实现后加固"的迭代方法：V1.0 快速完成功能原型，V2.0 针对安全测试发现的问题逐一修复。这种迭代方式完整模拟了真实项目的安全加固流程。

## 二、V1.0 实现

### 2.1 架构设计

```
templates/
├── base.html      → 骨架（导航栏 + 内容容器）
├── login.html     → 登录表单
└── index.html     → 用户信息页

static/css/
└── style.css      → 全局样式

app.py             → 路由 + 用户字典 + Session
```

Jinja2 模板继承：`login.html` 和 `index.html` 通过 `{% extends "base.html" %}` 共享导航栏和页面框架。

### 2.2 核心代码

```python
app = Flask(__name__)
app.secret_key = "dev-key-2025"

USERS = {
    "admin": {"password": "admin123", "role": "admin", ...},
    "alice": {"password": "alice2025", "role": "user", ...}
}

@app.route("/login", methods=["GET", "POST"])
def login():
    if request.method == "POST":
        username = request.form.get("username")
        password = request.form.get("password")
        if username in USERS and USERS[username]["password"] == password:
            session["username"] = username
            return render_template("index.html", user=USERS[username])
        else:
            error = "用户名或密码错误，请重试"
    return render_template("login.html", error=error)
```

### 2.3 前端设计

- 导航栏：`linear-gradient(135deg, #667eea, #764ba2)` 蓝紫渐变，flex 布局
- 卡片式布局：`border-radius: 8px`，`box-shadow` 阴影
- 表单：聚焦时紫色描边 `box-shadow: 0 0 0 2px rgba(102, 126, 234, 0.15)`

## 三、V1.0 安全测试

### 3.1 漏洞发现方法论

依据 OWASP Top 10 (2021) 分类，使用**攻击模拟**的方式逐项验证：对每个潜在攻击面设计攻击方案并执行，通过实际结果确认漏洞存在。

### 3.2 漏洞清单

| # | 漏洞名称 | CWE | 严重度 | 攻击难度 |
|---|---------|-----|--------|---------|
| 1 | HTML 注释泄露凭据 | CWE-312 | Critical | 极低 |
| 2 | 密码明文存储与直接比较 | CWE-256 | Critical | 低 |
| 3 | 登录后页面展示密码 | CWE-257 | High | 极低 |
| 4 | 无频率限制可暴力破解 | CWE-307 | High | 低 |
| 5 | 弱 secret_key 可伪造 Session | CWE-798 | Critical | 中 |
| 6 | 无 CSRF 保护 | CWE-352 | High | 中 |
| 7 | HTTP 明文传输 | CWE-319 | Medium | 中 |

### 3.3 漏洞验证与原理分析

#### 漏洞 1：HTML 注释泄露 (CWE-312)

`templates/login.html` 第 1 行：

```html
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```

```bash
curl -s http://192.168.137.135:5000/login | head -1
# 输出：<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```

HTML 注释不参与 DOM 渲染，但会完整暴露在页面源码中。这是典型的敏感信息泄露反模式。

#### 漏洞 2：密码明文存储与直接比较 (CWE-256)

```python
"password": "admin123"                                  # 明文存储
if USERS[username]["password"] == password:              # 直接比较
```

代码可"看到"所有密码原文。任何能读代码的人（机房同学、运维人员、代码泄露）都能获取全部用户密码。

**知识点**——为什么不能用 MD5/SHA256：
- MD5/SHA256 是快速哈希，GPU 每秒可算数十亿次
- 应使用慢哈希（bcrypt/scrypt/PBKDF2），通过大量迭代增加暴力破解成本
- Werkzeug 的 `generate_password_hash()` 默认使用 PBKDF2 + HMAC-SHA256，迭代 60 万次

#### 漏洞 3：登录后展示密码 (CWE-257)

```html
<span class="info-label">密码：</span>
<span class="info-value">{{ user.password }}</span>
```

登录后页面直接渲染密码明文，违反数据传输最小化原则。

#### 漏洞 4：无暴力破解防护 (CWE-307)

`/login` 路由无任何请求频率限制。

**攻击验证（ffuf）**：

```bash
ffuf -w passwords.txt -u http://127.0.0.1:5000/login \
     -X POST -d "username=alice&password=FUZZ" \
     -H "Content-Type: application/x-www-form-urlencoded" -fr "错误"
# alice2025  [Status: 200, Size: 2082]  ← 破解成功
```

**Hydra 验证**：

```bash
hydra -l alice -P passwords.txt 127.0.0.1 -s 5000 \
      http-post-form "/login:username=^USER^&password=^PASS^:错误"
# [5000][http-post-form] login: alice  password: alice2025
```

#### 漏洞 5：弱 secret_key + Session 伪造 (CWE-798)

```python
app.secret_key = "dev-key-2025"
```

Flask Session 的签名结构：

```
Cookie = Base64(JSON数据) + "." + 时间戳 + "." + HMAC-SHA1(secret_key, 数据.时间戳)
```

攻击者截获一个合法 Cookie，用字典爆破 secret_key 后即可伪造任意用户 session：

```python
for secret in wordlist:
    si = URLSafeTimedSerializer(secret, ...)
    try: si.loads(cookie); print(f"密钥: {secret}"); break   # 输出 dev-key-2025
    except: continue

si.dumps({'username': 'admin'})  # 无需密码，伪造 admin session
```

#### 漏洞 6：CSRF 跨站请求伪造 (CWE-352)

所有 POST 表单无 CSRF token。攻击者可构造钓鱼页面：

```html
<form action="http://192.168.137.135:5000/login" method="POST">
  <input name="username" value="admin">
  <input name="password" value="admin123">
</form>
<script>document.forms[0].submit()</script>
```

受害者访问此页面后，浏览器自动提交表单，将用户登录为 admin。攻击原理是**浏览器自动携带 Cookie**，而服务器不验证请求来源。

#### 漏洞 7：HTTP 明文传输 (CWE-319)

```bash
tcpdump -i any -A 'port 5000'
# POST /login → username=admin&password=admin123  （明文可见）
# Set-Cookie: session=eyJ1c2VybmFtZSI6...         （明文可见）
```

局域网内 ARP 欺骗 → 中间人嗅探 → 所有凭据和 Cookie 可被窃取。

## 四、V2.0 安全加固

### 4.1 加固方案

| 漏洞 | 修复方案 | 依赖 |
|------|---------|------|
| 注释泄露 | 删除，密码仅控制台打印 | — |
| 密码明文存储 | `generate_password_hash()` | werkzeug |
| 密码明文展示 | 过滤 password 字段后传模板 | — |
| 暴力破解 | 按 IP 限流 5次/分钟 | flask-limiter |
| 弱 secret_key | `os.urandom(24)` + 文件持久化 | os |
| CSRF | 全局 CSRF token | flask-wtf |
| Session 安全 | HttpOnly + SameSite=Lax | Flask 内置 |
| 安全响应头 | X-Content-Type-Options 等 | Flask 内置 |

### 4.2 各修复项实现

#### 4.2.1 密码哈希

```python
from werkzeug.security import generate_password_hash, check_password_hash

USERS = {
    "admin": {"password": generate_password_hash("Admin@2025#Secure"), ...},
    "alice": {"password": generate_password_hash("Alice@2025#Secure"), ...},
}

# 验证
if user and check_password_hash(user["password"], password):
    session["username"] = username
```

Werkzeug 输出格式：`pbkdf2:sha256:600000$salt$hash`，存储值自带算法、迭代次数和盐，验证时自动解析。

#### 4.2.2 限流

```python
from flask_limiter import Limiter
from flask_limiter.util import get_remote_address

limiter = Limiter(app, key_func=get_remote_address, storage_uri="memory://")

@app.route("/login", methods=["GET", "POST"])
@limiter.limit("5 per minute")
def login(): ...

@app.errorhandler(429)
def rate_limit_handler(e):
    return render_template("login.html", error="操作过于频繁，请 1 分钟后再试"), 429
```

#### 4.2.3 CSRF 保护

```python
from flask_wtf.csrf import CSRFProtect
csrf = CSRFProtect(app)
```

```html
<form method="POST" action="/login">
    <input type="hidden" name="csrf_token" value="{{ csrf_token() }}">
```

Token 绑定 session：攻击者无法获取受害者页面的 token，伪造请求被 400 拒绝。

#### 4.2.4 Session 安全

```python
app.secret_key = os.urandom(24).hex()       # 48 字符随机密钥
# 密钥写入文件，重启不丢失：
open(".secret_key", "w").write(app.secret_key)

app.config.update(
    SESSION_COOKIE_HTTPONLY=True,           # JS 不可读
    SESSION_COOKIE_SAMESITE="Lax",          # 跨站不发送
)
```

#### 4.2.5 安全响应头 + 密码过滤

```python
@app.after_request
def add_security_headers(response):
    response.headers["X-Content-Type-Options"] = "nosniff"
    response.headers["X-Frame-Options"] = "DENY"
    response.headers["X-XSS-Protection"] = "1; mode=block"
    return response

def get_safe_user(username):
    return {k: v for k, v in USERS[username].items() if k != "password"}
```

## 六、关键知识点

### 6.1 PBKDF2 密码哈希

```
DK = PBKDF2(PRF=HMAC-SHA256, Password, Salt=random(16), c=600000, dkLen=256bit)

固定参数工作流：
存储：user_input → PBKDF2(user_input, random_salt, 600000) → 存入 "算法$迭代$盐$哈希"
验证：user_input + 存储值 → 提取参数 → 重新计算 → 比较结果
```

600,000 次迭代是 OWASP 2024 推荐值。每次登录多耗 45ms，但暴力破解速度从"每秒十亿次"降至"每秒约 170 次"。

### 6.2 Flask Session 签名

```
Cookie = base64(payload) + "." + timestamp + "." + HMAC-SHA1(secret_key, payload.timestamp)
```

签名保证完整性（防篡改），但不加密（payload 可解码读取）。secret_key 是唯一防线——泄露即全部失守。

### 6.3 纵深防御

本次实践体现的防御层次：

```
Layer 1: 密码哈希          → 即使数据库泄露，密码不可逆
Layer 2: 限流              → 即使密码弱，每秒只能试几次
Layer 3: CSRF token        → 即使会话有效，不可跨站利用
Layer 4: SameSite Cookie   → 浏览器层面阻止跨站携带
Layer 5: 安全响应头         → 浏览器层面防注入/嗅探/劫持
Layer 6: 信息最小化        → 不展示、不传输敏感信息
```

## 七、总结

### 7.1 技术收获

1. **安全是设计属性而非后期补丁**：V1.0 每个模块都存在隐患，V2.0 证明安全因素需从架构层面融入
2. **Flask 生态的安全支持成熟**：Werkzeug、Flask-WTF、Flask-Limiter 将复杂安全机制封装为几行配置
3. **攻击工具可用作开发自检**：ffuf、Burp Intruder 是上线前验证防御有效性的有力手段
4. **计算不可行性构成安全基础**：安全的核心不是"不可能"，而是"代价远大于收益"
