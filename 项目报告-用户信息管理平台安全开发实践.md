## 第一步：写出能跑的 V1.0

### 我是怎么想的

老师说要做一个登录系统，我想的很简单：

1. 放个表单，让用户输入用户名和密码
2. 后端收到后去数据库里查——找到就登录成功，找不到就提示错误
3. 登录后显示用户信息
4. 加个登出按钮

听起来很直白，我就动手写了。

### Flask 是什么——我当时的理解

Flask 是一个 Python 的 Web 框架。我当时学到的几个关键概念：

**路由**：就是网址和函数的对应关系。比如用户访问 `/` 就执行 `index()` 函数，访问 `/login` 就执行 `login()` 函数。写法很简单：

```python
@app.route("/login")
def login():
    # 处理登录逻辑
```

**模板渲染**：Flask 用 Jinja2 把 Python 数据"填"进 HTML 页面。比如后端传一个 `user` 变量，前端就能用 `{{ user.username }}` 把用户名显示出来。我当时觉得这个特别好用。

**Session**：登录成功后怎么让服务器"记住"我？用 session。Flask 把 session 存在浏览器的 Cookie 里，每次请求浏览器自动带上，服务端就知道"哦，这是 admin"。

### 我写的第一版代码

这是核心的 app.py，当时我觉得写得挺好的：

```python
from flask import Flask, render_template, request, redirect, session

app = Flask(__name__)
app.secret_key = "dev-key-2025"          # 随便起个密钥

USERS = {
    "admin": {
        "username": "admin",
        "password": "admin123",          # 明文存密码
        "role": "admin",
        "email": "admin@example.com",
        "phone": "13800138000",
        "balance": 99999
    },
    "alice": {
        "username": "alice",
        "password": "alice2025",         # 也是明文
        "role": "user",
        "email": "alice@example.com",
        "phone": "13900139001",
        "balance": 100
    }
}

@app.route("/login", methods=["GET", "POST"])
def login():
    error = None
    if request.method == "POST":
        username = request.form.get("username", "")
        password = request.form.get("password", "")
        # 直接用 == 比较密码
        if username in USERS and USERS[username]["password"] == password:
            session["username"] = username
            user_info = USERS[username]    # 包含密码！
            return render_template("index.html", user=user_info)
        else:
            error = "用户名或密码错误，请重试"
    return render_template("login.html", error=error)
```

登录模板我还很贴心地写了个注释：

```html
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```

当时想的是"万一忘了密码可以回来看源码"——现在想想真是糊涂。

首页就更离谱了，登录后直接展示所有信息，**连密码都展示出来**：

```html
<div class="info-item">
    <span class="info-label">密码：</span>
    <span class="info-value">{{ user.password }}</span>  <!-- 密码直接显示 -->
</div>
```

## 第二步：学安全，开始攻击自己的代码

### 老师教我们用的第一个工具：curl

老师先在黑板上写了这个命令：

```bash
curl -s http://192.168.137.135:5000/login | head -1
```

运行结果吓了我一跳：

```
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```

**我的反应**："就这样拿到了？连登录都不用？"

我突然明白了：网页上虽然看不到注释，但任何查看网页源代码的人都能看到。如果有人在机房用班里的公共电脑访问这个系统，右键一点就能看到管理员密码——太可怕了。

### 发现第二个问题：密码是明文的

老师接着让我们看 app.py 的第 45 行：

```python
if USERS[username]["password"] == password:
```

"这里用了 `==` 直接比较"，老师解释说，"这意味着代码能直接'看到'密码原文。如果有人通过某种方式拿到了代码文件，或者入侵了服务器，所有用户的密码就全部暴露了。"

我当时还不太理解："那不用 `==` 用什么？不用 `==` 怎么判断密码对不对呢？"

老师没有直接回答，让我们先继续找漏洞。这个问题我在后面修复时才真正搞明白。

### curl 解决不了的问题——老师教我们用 Burp Suite

curl 只能看源码，但不能"拦截"和"修改"请求。老师介绍了 Burp Suite。

我当时最迷惑的就是——"为什么要设置浏览器的代理服务器？为什么要监听 127.0.0.1:8080？"

老师打了个比方：**就像快递中转站**。

正常情况下，你的浏览器直接给目标服务器发请求，Burp 根本看不到。设置代理后，浏览器先把请求送到 Burp（127.0.0.1:8080），Burp 检查/记录/修改后，再帮你转发到目标服务器。

```
没设代理： 浏览器 ────────→ Flask服务器
设了代理： 浏览器 → Burp(127.0.0.1:8080) → Flask服务器
                         ↑
                    拦截、查看、修改
```

`127.0.0.1` 就是"本机自己"的意思，`8080` 是 Burp 的默认门牌号。所以设置代理 = 告诉浏览器"以后寄快递先送到中转站"。

### 用 Burp 拦截登录请求

配置好代理后，我打开 login 页面，填了用户名和密码，点登录——Burp 马上拦截到了请求：

```
POST /login HTTP/1.1
Host: 192.168.137.135:5000

username=alice&password=test123
```

我右键点 "Send to Intruder"，这步是接下来做字典攻击的关键。

### 学习字典攻击（暴力破解）

老师让我们准备一个密码文件，里面放一批常用密码：

```
123456
password
admin
alice
alice123
alice2024
alice2025      ← 这个是真的
admin123
```

然后打开 Burp 的 Intruder 模块：

1. 在密码的值上面标记 `§` 符号：`password=§test123§`
2. 在 Payloads 里加载密码字典
3. 在 Options 里设置 Grep-Match，搜索 "欢迎回来"（表示登录成功）
4. 点击 "Start attack"

结果出来了：

```
#1 123456    → 200  没有"欢迎回来"
#2 password  → 200  没有"欢迎回来"
...
#7 alice2025 → 200  ✅ 有"欢迎回来"！
```

第 7 次就找到了密码！如果有 10000 个密码的字典，也就几分钟的事。

老师还教我们用命令行工具 ffuf，更快：

```bash
ffuf -w passwords.txt -u http://127.0.0.1:5000/login \
     -X POST -d "username=alice&password=FUZZ" \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -fr "错误"
```

这个工具不到 1 秒就跑完了 20 个密码的字典。Hydra 也是类似的效果。

### 发现了更多漏洞

接下去老师带着我们一个一个找：

**Session Cookie 可以被伪造**——因为我们用了弱密钥 `dev-key-2025`。老师教我们用 Python 写了个爆破脚本，遍历一个简单字典，几轮就对出了密钥。有了密钥，就能伪造任意用户的 Cookie，不需要密码就能冒充 admin。

**没有 CSRF 保护**——攻击者可以做一个钓鱼页面，里面藏一个自动提交的表单。受害者只要点开页面，浏览器就会自动以 admin 的身份登录。原理是浏览器发请求时会自动带上 Cookie，而服务器不会验证"这个请求是从哪个页面发来的"。

**流量是明文的**——用 tcpdump 抓包一看，POST 请求里的 `username=admin&password=admin123` 清清楚楚，Cookie 也明文可见。校园网内任何一台设备如果进行了 ARP 欺骗，就能抓到所有人的密码。

### 我用表格整理了一下我找到的所有漏洞

| 漏洞 | 我一开始怎么做的 | 问题在哪 | 严重程度 |
|------|---------------|---------|---------|
| HTML注释泄露密码 | 把 admin/admin123 写在注释里 | 任何人右键查看源码就能拿到 | 🔴 最严重 |
| 密码明文存储 | `"password": "admin123"` | 泄露代码 = 泄露所有密码 | 🔴 最严重 |
| 用 `==` 比较密码 | `password == stored` | 同上，系统需要"看到"明文 | 🔴 最严重 |
| 登录后显示密码 | 模板里 `{{ user.password }}` | 任何登录的人都能看到自己密码 | 🟠 高危 |
| 没有限流 | 登录接口随便调 | ffuf/hydra 1秒就能爆出密码 | 🟠 高危 |
| secret_key 太弱 | `"dev-key-2025"` | 被爆破后可伪造任意用户 session | 🔴 最严重 |
| 没有 CSRF token | 表单直接 POST | 可以被钓鱼页面利用 | 🟡 中危 |
| HTTP 明文传输 | 没配 HTTPS | 局域网嗅探可获密码和 Cookie | 🟡 中危 |

说实话，找到这些漏洞后我挺沮丧的。好不容易写出来的东西，居然全身都是漏洞。但老师说得对："**如果连自己造的漏洞都攻击不了，怎么防御别人的攻击？**"这话点醒了我。

## 第三步：修复所有漏洞，做出 V2.0

知道了问题在哪，就该修了。下面记录我的修复过程，以及每一步背后学到的原理。

### 修复 1：删掉 HTML 注释，密码只在控制台显示

这个最简单——把那行注释删了就行。但我又想：那怎么知道默认密码呢？于是把账号密码改到启动时在终端打印出来：

```python
print("=" * 56)
print("  默认账号       密码")
print("  admin          Admin@2025#Secure")
print("  alice          Alice@2025#Secure")
print("=" * 56)
```

这样调试信息只在服务器本地终端可见，前端完全拿不到。

### 修复 2：密码哈希——这次我终于搞懂了

前面我问过"不用 `==` 怎么验证密码？"——答案就是：**哈希函数**。

**哈希函数的特点是单向的**：你输入一个密码，它输出一串乱码。但你没法从乱码反推密码。

但问题来了——如果用同样的输入，哈希输出也是一样的。比如两个人的密码都是 `123456`，哈希后也相同，攻击者可以提前算好常见密码的哈希值做成"彩虹表"来查。

解决方法是**加盐（Salt）**——给每个密码随机配一段值混进去再哈希，这样即使两个人密码一样，哈希结果也不同。

Flask 自带的 Werkzeug 库已经封装好了这些，我只需要：

```python
from werkzeug.security import generate_password_hash, check_password_hash

# 存储时
USERS["admin"]["password"] = generate_password_hash("Admin@2025#Secure")
# 结果类似: pbkdf2:sha256:600000$随机盐$哈希值

# 验证时
if check_password_hash(USERS[username]["password"], user_input):
    # 密码正确
```

`check_password_hash` 内部会：
1. 从存储的值里提取加密算法、盐、迭代次数
2. 用同样的参数对输入密码做哈希
3. 比较结果

**关键理解**：验证的时候不需要知道原始密码，只需要知道"同样的输入 + 同样的盐 + 同样的算法 = 同样的输出"，能对上就说明密码正确。

而且 Werkzeug 用的是 PBKDF2 算法，默认迭代 **60 万次**。正常登录多花几十毫秒完全无所谓，但对攻击者来说，每个密码都要算 60 万次哈希，暴力破解的速度被大大降低了。

### 修复 3：不把密码传给模板

修复很简单，写一个过滤函数：

```python
def get_safe_user(username):
    user = USERS.get(username)
    if user:
        return {k: v for k, v in user.items() if k != "password"}
    return None
```

用字典推导式把 password 字段去掉，这样模板里永远拿不到密码。

### 修复 4：加上限流——让攻击者每秒最多试 5 次

用了 Flask-Limiter 库：

```python
from flask_limiter import Limiter
from flask_limiter.util import get_remote_address

limiter = Limiter(
    app=app,
    key_func=get_remote_address,
    default_limits=["200 per day", "50 per hour"],
)

@app.route("/login", methods=["GET", "POST"])
@limiter.limit("5 per minute")   # 只对登录接口加强限制
def login():
    ...

@app.errorhandler(429)
def rate_limit_handler(e):
    return render_template("login.html",
        error="操作过于频繁，请 1 分钟后再试"), 429
```

我测试了一下：连续发 5 次错误的登录请求没问题，到第 6 次就返回 429（Too Many Requests），页面显示 "操作过于频繁，请 1 分钟后再试"。

**这意味着什么**？之前 ffuf 一秒能试几百个密码，现在每 12 秒才能试一个。如果密码有 8 位大小写字母+数字组合，破解时间从几分钟变成了上千年。

### 修复 5：用安全的随机密钥

```python
import os
app.secret_key = os.urandom(24).hex()
# 生成类似: "e8f3a9b2c0d7..." 的64位随机字符串
```

之前 `dev-key-2025` 能被字典爆破，现在改成 24 字节的随机值（相当于 48 位十六进制字符串），暴力破解需要约 2^192 次尝试，在计算上不可行。

另外还加了个 Smart 处理：把密钥写入文件，重启时读取，这样用户不会因为服务器重启而被登出。

### 修复 6：CSRF 保护

用 Flask-WTF 库。在 app.py 里加一行：

```python
from flask_wtf.csrf import CSRFProtect
csrf = CSRFProtect(app)
```

然后在登录表单里加一个隐藏字段：

```html
<input type="hidden" name="csrf_token" value="{{ csrf_token() }}">
```

**原理**：每次用户访问登录页时，服务端生成一个随机 token，嵌在表单里。提交时一起发回服务器验证。攻击者就算构造了钓鱼页面，也拿不到这个 token（因为 token 跟用户的 session 绑定），所以恶意提交会被拒绝（400 Bad Request）。

我实际测试了一下：用 curl 直接 POST 不带 token 的请求，确实被拒绝了。

### 修复 7：给 Cookie 加上安全标志

```python
app.config.update(
    SESSION_COOKIE_HTTPONLY=True,   # JS 代码读不到 Cookie
    SESSION_COOKIE_SAMESITE="Lax",  # 跨站请求不带 Cookie
)
```

### 修复 8：加上安全响应头

```python
@app.after_request
def add_security_headers(response):
    response.headers["X-Content-Type-Options"] = "nosniff"      # 防 MIME 嗅探
    response.headers["X-Frame-Options"] = "DENY"                 # 防被嵌入 iframe
    response.headers["X-XSS-Protection"] = "1; mode=block"       # 浏览器 XSS 过滤
    return response
```

### 验证：用同样的攻击手段再打一次

| 攻击手段 | V1.0 结果 | V2.0 结果 |
|---------|---------|---------|
| 看网页源码找密码 | ✅ 直接拿到 | ❌ 什么都没有 |
| ffuf 字典爆破 | ✅ 1秒破解 | ❌ 第6次就被 429 拦截 |
| 伪造 session cookie | ✅ 用弱密钥直接伪造 | ❌ 随机密钥，不可爆破 |

## 学到的知识整理

### 新学到的核心概念

| 概念 | 一句话解释 | 我之前不知道 |
|------|-----------|-------------|
| **密码哈希** | 单向函数，存哈希不存原文 | 以为存数据库就行 |
| **加盐（Salt）** | 每个密码加随机值再哈希 | 不知道彩虹表攻击 |
| **PBKDF2** | 迭代 60 万次的慢哈希 | 以为 MD5 就够了 |
| **CSRF** | 利用登录态跨站发请求 | 完全没听过 |
| **Session 签名** | Cookie = 数据 + HMAC签名 | 以为 Cookie 是加密的 |
| **限流** | 按 IP 限制请求频率 | 以为只有大厂才需要 |
| **SameSite Cookie** | 跨站请求不发送 Cookie | 不知道这个属性 |
| **安全响应头** | 浏览器遵循的安全指令 | 不知道能防什么 |

### 学到的工具

| 工具 | 干什么用 | 怎么用的 |
|------|---------|---------|
| **curl** | 命令行发 HTTP 请求 + 看源码 | 发现注释泄露密码 |
| **Burp Suite** | HTTP 代理拦截 + 字典攻击（Intruder） | 拦截登录请求 → 标记攻击位置 → 加载字典 → 爆破 |
| **ffuf** | 命令行快速模糊测试 | `ffuf -w 字典 -u URL -d "password=FUZZ"` |
| **Hydra** | 专业登录爆破工具 | `hydra -l 用户名 -P 字典 IP http-post-form` |
| **tcpdump** | 抓包看网络流量 | 看到 HTTP 明文传输密码和 Cookie |
| **git** | 版本控制 + GitHub 托管 | 记录代码迭代，展示项目成果 |

### 让我印象最深的安全原则

1. **安全不是事后打补丁的**——V1.0 的代码几乎每行都有隐患。如果一开始就考虑安全设计，后面不用改那么多。

2. **"慢"就是安全**——PBKDF2 的 60 万次迭代让登录多了几十毫秒，但这几十毫秒让暴力破解从"几分钟"变成了"上百年"。

3. **攻击自己的代码是最好的老师**——看理论说"明文密码不安全"没什么感觉，亲手用 ffuf 攻破自己的登录，记忆才刻骨铭心。

4. **层层设防（纵深防御）**——单靠一个措施不够。密码哈希 + 限流 + CSRF token + 安全响应头，每一层都在增加攻击成本。

## 我的感悟

### 这个项目让我改变的地方

**从"代码能跑就行"变成"代码安全了吗"**。写 V1.0 的时候我只关心功能对不对，写 V2.0 的时候每加一行代码都会想：这里会不会有注入？这个数据要不要过滤？用户能看到什么？

**工具不是黑客专属，开发者更应该用**。以前觉得 Burp Suite 是黑客工具，离我很远。现在明白了——开发者也应该用它，在上线前先自己攻击自己一遍，发现问题就修，比被真正的攻击者发现好千万倍。

**Flask 的生态帮了大忙**。如果让我从零实现密码哈希、CSRF 保护、限流，我根本不知道从何下手。但 Werkzeug、Flask-WTF、Flask-Limiter 这些库已经把最难的部分封装好了，我只需要理解原理、正确配置就行。

### 最骄傲的时刻

不是写完 V2.0 的时候，而是 **V2.0 上线后用 ffuf 攻击失败了** 的时候。因为那一刻我知道——我写的防御真的起作用了。
